    3Ô∏è‚É£ Embedding vs. References

    üìç Key Considerations

    ‚úÖ Embedding ‚Üí Store data together for efficiency.
    ‚úÖ Referencing ‚Üí Normalize data for scalability and flexibility.

    • Data that is accessed together should be stored together.
    • Is the app read-heavy or write-heavy?
      • Read-heavy ‚Üí Consider embedding.
      • Write-heavy ‚Üí Consider referencing to avoid document growth issues.

    üìå When to Embed?

    ‚úÖ Use embedding when: - Simplicity is preferred. - There is a "has-a" or "contains" relationship. - Data is frequently retrieved together. - Updates should be atomic. - Minimal data duplication is required. - The document size is manageable (MongoDB 16MB limit). - Child cannot exist without the parent.

    ‚ùå Avoid embedding if: - Large amounts of embedded data cause document bloat. - Data duplication makes updates hard to manage. - Children can exist independently (use references instead).

    üìå One-to-One Relationships

    ‚úÖ When to Embed?

    • If both entities are always queried together.
    • Example: User Profile ‚Üí User Settings (single document).
    {
  "_id": ObjectId("123"),
  "name": "John Doe",
  "settings": {
    "theme": "dark",
    "notifications": true
  }
}

    ‚úÖ When to Reference?

    • When the child object may exist independently.
    • Types of referencing:
      • One object has a foreign key of the other.The most accessed document should store the foreign key.
      • Both reference each other with foreign keys.

    {
  "_id": ObjectId("123"),
  "name": "John Doe",
  "profile_id": ObjectId("456")
}

    {
  "_id": ObjectId("456"),
  "user_id": ObjectId("123"),
  "bio": "Software Developer"
}

    üìå One-to-Many Relationships

    ‚úÖ When to Embed?

    • Few child documents per parent.
    • Data is always queried together.
    • Example: Blog ‚Üí Comments.
    {
  "_id": ObjectId("123"),
  "title": "MongoDB Best Practices",
  "comments": [
    { "user": "Alice", "text": "Great article!" },
    { "user": "Bob", "text": "Thanks for sharing." }
  ]
}

    ‚úÖ When to Reference?

    • If the child collection grows indefinitely.
    • Types of referencing:
      • Parent or children has the foreign key of each other.
      • Both parent and children have foreign key.
    • Example: User ‚Üí Orders.
    {
  "_id": ObjectId("123"),
  "name": "John Doe",
  "orders": [ ObjectId("order1"), ObjectId("order2") ]
}

    {
  "_id": ObjectId("order1"),
  ...
}

    üìå Many-to-Many Relationships

    ‚úÖ When to Embed?

    • Data duplication is acceptable.
    • Example: Author ‚Üí Books (Embedded List).
    {
  "_id": ObjectId("123"),
  "name": "J.K. Rowling",
  "books": [
    { "title": "Harry Potter 1" },
    { "title": "Harry Potter 2" }
  ]
}

    ‚úÖ When to Reference?

    • Avoid bidirectional references (expensive joins).
    • Use an intermediate "junction" collection.
    • Example: Students & Courses (Many-to-Many Reference).
    // Students Collection
{
  "_id": ObjectId("s1"),
  "name": "Alice",
  "enrolled_courses": [ ObjectId("c1"), ObjectId("c2") ]
}

// Courses Collection
{
  "_id": ObjectId("c1"),
  "title": "MongoDB 101",
  "students_enrolled": [ ObjectId("s1"), ObjectId("s2") ]
}

// Junction Collection (Enrollment)
{
  "_id": ObjectId("e1"),
  "student_id": ObjectId("s1"),
  "course_id": ObjectId("c1"),
  "enrollment_date": "2024-03-08"
}

    Use Case Embedding ‚úÖ Referencing ‚úÖ

    One-to-Many ‚úîÔ∏è If small & accessed together ‚úîÔ∏è If array is unbounded
    Many-to-Many ‚ùå Not recommended ‚úîÔ∏è Use an intermediate collection
